# Init Function Translation Issue â€” Context Document

## The Problem

During DBEMT Phase D, we translated DBEMT_Init (127 lines) to C++. This exposed a fundamental architectural problem: **init functions allocate their INTENT(OUT) types, but C++ can't do Fortran ALLOCATE.** VIT's standard wrapper (populate views â†’ call C++ â†’ reverse-copy) crashes because the INTENT(OUT) types are unallocated at function entry â€” `vit_populate` calls `C_LOC` on unallocated arrays â†’ segfault.

## What We Tried

### Attempt 1: Hand-written Fortran wrapper
Manually wrote a ~60-line Fortran wrapper that duplicates the original function's allocation logic, then calls C++ for the ~25 lines of computation (set scalars, compute spanRatio). This worked (3/3 KGen, 17/17 baselines) but was fragile â€” the wrapper duplicates logic, can't be auto-regenerated by VIT, and broke the integrate_all.sh automation pattern.

### Attempt 2: VIT's --auto-allocate flag
Discovered that VIT already had `--auto-allocate` infrastructure: `allocate_extract.py` parses ALLOCATE statements from Fortran source, `interface_gen.py` emits them in the wrapper before view populate. Three gaps were fixed:
1. Conditional allocation guards (IF conditions around ALLOCATEs)
2. Error handling (STAT= checks with error calls after each ALLOCATE)
3. Copy-call detection (CALL statements that allocate via deep copy)

This eliminated the hand-written wrapper. `integrate_all.sh` uses `--auto-allocate` and the full cycle works (25/25 integrate, 17/17 baselines).

### Attempt 3: Audit revealed deeper issues
Code audit found that `--auto-allocate` works by regex-matching Fortran coding patterns (ALLOCATE followed by IF/error-call/RETURN/END IF, DO loops wrapping Copy calls, specific error variable naming). These patterns are coding-style-specific, not Fortran-language-specific. A different codebase would structure init functions differently and the extraction would fail.

We fixed several issues (guard nesting, copy-call arg order, verbatim error block extraction), but the fundamental fragility remains: **--auto-allocate parses imperative code style, not language constructs.**

## The Realization

Init functions should NOT be translated during the incremental phase. They should stay in Fortran and only be translated directly to pure C++ when the module converts.

**Why init functions are different from runtime functions:**
- Run once at startup â€” zero performance benefit from C++
- Primarily allocation + scalar setup â€” trivial computation
- Already call translated functions via Fortran wrappers â€” this works fine
- The C++ "translation" of DBEMT_Init is 25 lines. The wrapper is 60 lines. Net negative.

**When init functions SHOULD be translated:**
When converting a module to pure C++. At that point:
- ALLOCATABLE becomes `new`/`malloc`/`std::vector`
- No Fortran wrapper exists â€” C++ does everything
- The allocation problem disappears

### Task 4: Build caller dependency tracking (wrapper elimination roadmap)

**Why this matters:** Every translated function has a Fortran wrapper that exists because its CALLERS are still Fortran. The wrapper is the bridge â€” it provides the Fortran subroutine interface that callers expect, forwards to C++. The wrapper can ONLY be removed when ALL callers of that function are also in C++ (and call the `_c` function directly).

**What we need to track per translated function:**
- Which Fortran functions call it (the callers)
- Whether each caller is translated or still Fortran
- When the last Fortran caller is translated â†’ wrapper eligible for removal

**Example:** `DBEMT_CalcContStateDeriv` is called by:
- `DBEMT_RK4` (translated âś“)
- `DBEMT_AB4` (translated âś“)
â†’ All callers translated â†’ wrapper could be removed (both callers could call `dbemt_calccontstatederiv_c` directly)

**Example:** `AFI_ComputeAirfoilCoefs` is called by:
- `BEMT_CalcOutput` (NOT translated âś—)
- `UA_CalcOutput` (NOT translated âś—)
â†’ Wrapper must stay until BEMT and UA are translated

**Data source:** `query_coverage.py --call-sites FunctionName` already returns all call sites with file/line/function context. This can be extended to cross-reference against vit.yaml's translation list.

**Implementation options:**
1. **vit.yaml section per function:** Add `callers:` list with status. Manual maintenance â€” fragile.
2. **VIT command:** `vit status --wrappers` that auto-scans call sites and cross-references against translations. Shows which wrappers are ready for removal. Best option â€” derived from source, always current.
3. **Separate tracking file:** A generated report showing the dependency graph. Good for planning, but stale if not regenerated.

**The dependency chain for wrapper elimination:**
- Leaf functions called only by other translated functions â†’ wrappers removable now
- Functions called by untranslated callers â†’ wrappers stay
- Top-level entry points (AD_Init, AD_CalcOutput) â†’ wrappers stay until OpenFAST glue code is C++
- Init functions â†’ stay in Fortran entirely until module converts to pure C++

**This tracking enables module-by-module conversion planning:** When all functions in a sub-system AND all their callers are translated, that sub-system can go pure C++ (remove wrappers, remove Fortran module, link C++ directly).
